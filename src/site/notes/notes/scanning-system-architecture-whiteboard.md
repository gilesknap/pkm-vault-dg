## Overview

Whiteboard session diagramming a motion/scanning control system, covering the full pipeline from scan specification through trajectory planning to hardware execution. Captured 2026-06-02.

## 1 · Architecture & Data-Flow (top panel)

The top section shows an end-to-end data-flow in teal/black marker.

- **[[UIT\|UIT]]** is the upstream scan-instruction source.
- **[[Scanspec\|Scanspec]]** compound wraps the spec with a minimum-movement guard of 1.
- The **BL model** contains the virtual→real array transformation chain, invoking kinematics before passing real arrays to a GPU-assisted check stage.
- A **Queue** receives an `id`; a circled annotation reads *"uncomfortable id"* — likely a design concern about id propagation.
- **[[Bueapi\|Bueapi]]** consumes the queue output and feeds back `pos`, `Live`, `padua`, and `pcomp` signals alongside raw arrays.
- `from id → [[BLState\|BLState]]` is shown at top-right, indicating state is derived from the queue id.

```
UIT
 └─► Scanspec Compound (min_movement = 1)
       ├─ [no_check path]
       └─► BL model
             ├─ scanspec (virtual)
             │    └─ iter → virtual arrays (repeat if stashed)
             ├─ run kinematics
             └─ real arrays → checks ←── GPU
                                  │
                                  ▼
                               Queue ◄── id
                                  │
                                  ▼
                               Bueapi
                    (pos, Live, padua, pcomp / arrays feedback)
```
